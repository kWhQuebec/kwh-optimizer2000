# Audit complet — kWh Optimizer 2000
**Date** : 6 mars 2026
**Audité par** : Claude (navigation live + revue de code)
**Scope** : UX/UI, hypothèses financières, calculs, design, architecture de données

---

## RÉSUMÉ EXÉCUTIF

L'application est fonctionnelle et couvre un workflow solaire C&I complet (exploration → opération). Le design est professionnel. Cependant, plusieurs problèmes critiques affectent la crédibilité des résultats présentés aux clients et bloquent la progression du workflow.

**3 problèmes critiques** :
1. Le système recommande des tailles PV qui dépassent la capacité du toit (50 kWc recommandé sur un toit de 8 kWc)
2. 14 tâches du workflow sont hardcodées `false` → Step 5-6 ne peuvent jamais se compléter
3. Aucune validation croisée entre taille PV, surface de toit, et PMA du bâtiment

---

## 1. BUGS & PROBLÈMES CRITIQUES

### 1.1 PV > Capacité toit (CRITIQUE)
- **Constat** : Sur le site "1350 Chomedey", le système recommande 50 kWc mais affiche "Capacité toit estimée: 8.0 kWc — Limité par le toit"
- **Cause** : `design.tsx:58` — `pvSizeKW: z.coerce.number().min(0)` sans max. Aucune validation croisée avec `roofAreaSqM`
- **Impact** : Résultats financiers présentés au client sont basés sur une taille impossible à installer
- **Fix** : Ajouter validation `pvSizeKW <= (roofAreaSqM * roofUtilization) / moduleAreaPerKW`

### 1.2 Workflow Steps 5-6 bloqués (CRITIQUE)
- **Constat** : 14 détecteurs dans `useWorkflowProgress.ts` retournent `() => false` en permanence
- **Tâches affectées** : Toutes les tâches de construction (Step 5) et opération (Step 6)
- **Impact** : Un projet ne peut jamais être marqué "complété". Le tableau de bord affiche 0 kW installés même après livraison
- **Fix** : Implémenter le tracking d'activités pour les étapes construction + O&M

### 1.3 Pages en chargement permanent
- **Constat** : Analyses, Designs, et Tâches à faire affichent des skeletons qui ne se résolvent jamais (observé en navigation live)
- **Code** : Le code a des fallbacks correctes (empty state si `isLoading=false`), donc c'est probablement un **problème d'API** — la requête `/api/simulation-runs` ou `/api/designs` ne retourne jamais
- **Fix** : Ajouter un timeout + état d'erreur explicite. Vérifier les endpoints côté serveur

### 1.4 Route /dashboard → 404
- **Constat** : Cliquer "Tableau de bord" depuis le 404 redirige vers `/app` (correct), mais `/dashboard` en direct donne 404
- **Cause** : Aucune route `/dashboard` dans `App.tsx`. Seul `/app` existe
- **Fix** : Ajouter redirect `/dashboard` → `/app` ou supprimer tout lien vers `/dashboard`

## 3. HYPOTHÈSES FINANCIÈRES — AUDIT

### 3.1 Valeurs par défaut actuelles
| Paramètre | Valeur | Réaliste? | Commentaire |
|-----------|--------|-----------|-------------|
| Taux d'actualisation (WACC) | 7% | ✅ | Standard C&I Québec |
| Inflation tarif HQ | 3.5% | ⚠️ | Historiquement 2.5-3%, 3.5% est optimiste (favorable au solaire) |
| Taux d'imposition | 26.5% | ✅ | Fédéral + provincial correct |
| Dégradation modules | 0.4%/an | ✅ | PVSyst-validated, conservateur |
| CAPEX solaire | $2.25/W | ⚠️ | Correct pour <100 kW, mais trop élevé pour >500 kW (devrait être ~$1.60-1.80/W) |
| Coût batterie | $450/kWh | ✅ | Correct 2025-2026 pour LFP C&I |
| O&M solaire | 1% CAPEX | ✅ | Standard industrie |
| Compensation surplus HQ | $0.046/kWh | ✅ | Tarif avril 2025 confirmé |
| Incitatif HQ solaire | $1000/kW, max 40% | ✅ | OSE 6.0 confirmé |
| CII fédéral | 30% | ✅ | Correct pour 2025-2026 |
| Bouclier fiscal (DPA) | 90% × taux impôt | ⚠️ | Le facteur 90% est approximatif — devrait utiliser le CCA réel classe 43.1/43.2 |
| Rendement solaire | 1150 kWh/kWp | ✅ | Réaliste pour le sud du Québec |
| Marge EPC | 35% | ⚠️ | Élevé — standard industrie 20-25%. À 35%, les paybacks sont gonflés |
| Ratio ILR (DC/AC) | 1.45 | ✅ | Standard optimisé |

### 3.2 Problèmes de calcul identifiés
1. **CAPEX non dégressive** : Le coût de $2.25/W est fixe peu importe la taille. Un projet de 500 kW devrait avoir un coût significativement plus bas (~$1.60/W). Impact : surévaluation du CAPEX de 25-35% sur les gros projets
2. **Inflation HQ à 3.5%** : Favorise le solaire dans les projections 25 ans. Si HQ applique seulement 2.5%, les économies cumulées sont surévaluées de ~15%
3. **Marge EPC 35%** : Si c'est appliqué au prix client, ça gonfle le payback. Si c'est la marge interne, c'est distinct. Clarifier l'utilisation
4. **Pas de courbe d'apprentissage** sur le CAPEX batterie : Le remplacement à l'année 10 utilise un facteur de 60% du coût initial, mais le marché pourrait baisser plus vite
5. **Bouclier fiscal approximatif** : Le calcul `90% × net CAPEX × 26.5%` simplifie la réalité du CCA classe 43.1 (30% dégressif) et 43.2 (50% dégressif). Devrait modéliser année par année

## 6. ARCHITECTURE — SUGGESTIONS STRUCTURELLES

### Court terme
- **Externaliser les hypothèses financières** dans une table `financial_defaults` en DB au lieu de les hardcoder dans `schema.ts`. Permet de les modifier sans déploiement
- **Ajouter des contraintes FK** manquantes (users.clientId → clients.id)
- **Timeout + retry** sur les requêtes API longues (React Query `staleTime`, `retry`)

### Moyen terme
- **Multi-tenant propre** : Actuellement l'admin voit tous les clients/sites. Ajouter des rôles "account manager" qui ne voient que leurs projets assignés
- **Webhooks/notifications** : Quand un site passe à une étape suivante, notifier l'account manager par email
- **API publique** : Pour permettre l'intégration avec des outils externes (CRM, comptabilité)
